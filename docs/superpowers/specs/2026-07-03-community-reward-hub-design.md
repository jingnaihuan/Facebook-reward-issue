# 社群互动发奖中台 — 设计文档

日期：2026-07-03
作者：石上 / Claude

## 目标

把「社群互动活动发奖」这条固定但繁琐的流程，收进一个本地网页中台，一站式走完：
抓留言 → 提取玩家 ID → 拉玩家信息 → 按语言筛选 → 按发奖规则出名单 → 导出。
替代现在「FB 抓取网页 + Excel VBA + 手工 Eastblue 下载 + 手工分 sheet」的多工具拼接。

## 核心流程（漏斗顺序）

**关键：发奖规则只在「语言筛选后的最终名单」上运行。**

```
① 抓全部留言 + 回复（FB Graph API，浏览器端直连）
      ↓
② 提取玩家 ID（VBA 同款正则）+ 去重
      ↓
③ 拉 Eastblue 玩家信息，按 ID 匹配
      ↓
④ 按语言筛选（活动面向 en 就只留 en 玩家）★ 剔除的进「无效」名单
      ↓
⑤ 发奖规则（前 N 楼 / 随机抽 N / 点赞最高 / 回复最高）只作用于 ④ 的结果 ★
      ↓
⑥ 导出（特殊奖 sheets + 参与奖 sheet + 无效 sheet）
```

举例：一个帖子下有 de 和 en 玩家回复，但活动只面向 en。那么「前 100 楼」= **筛掉 de 之后、en 玩家里的前 100 楼**；「随机抽 5 名」也只在这批 en 玩家里抽。非目标语言玩家在进规则引擎前就已剔除。

## 架构

**方案 A：本地网页中台**（复用 LocoFlow 本地工具的「backend + frontend + 一键启动」模式）

- **前端**：单页五步向导（浏览器）。FB 抓留言在前端直连 `graph.facebook.com`（整合现有 `Projects/Facebook留言抓取/index.html` 的登录 + 抓取逻辑，公司网络已验证可行，回复也能抓到）。
- **后端**：本地 Python 服务。负责 Eastblue 自动下载、xlsx 解析、规则引擎计算、导出。
- **一键启动**：`启动.command`（Mac 优先），先跑通 Mac 再考虑 Windows。

**为什么要本地后端**：Eastblue 导出链接是网页地址（`#/` 前端路由 + `auto_download=1`），下载由前端 JS 触发，程序无法直接 GET 到文件。所以由后端用 **Playwright** 打开链接、等 xlsx 自动落盘到指定目录、再读取。浏览器安全策略下纯前端网页也无法自动读取用户下载目录里的文件，因此需要一个能读文件、跑 Playwright、处理 xlsx 的本地进程。

**目录结构**：
```
Projects/社群互动发奖中台/
  backend/        # Python：FastAPI/Flask + 规则引擎 + Eastblue 下载 + xlsx 读写
  frontend/       # 单页向导
  启动.command    # Mac 一键启动
  docs/
  测试用文档/     # 回归测试用真实数据
```

## 五步向导（前端）

### 步骤 1 · 抓取留言
- 整合现有 FB 抓取工具：保留「方式 A App ID 登录」「方式 B 贴 Token」两种。
- 选粉专 → 选帖子 → 抓全部留言 + 回复（含点赞数、回复数、留言时间、留言顺序）。
- 兜底：支持直接导入 CSV / Excel（列：序号、名称、留言内容、按赞数、留言时间），用于已有原始留言的场景。

### 步骤 2 · 清洗提取 ID
- 正则 `(?:^|\D)(1\d{9})(?!\d)`（与 VBA `GetID` 完全一致：1 开头 10 位，前后都不是数字，避免 11 位误判）。
- 表格实时预览：绿 = 提取成功；红 = 未找到合规 ID（自动归入无效名单）。
- **去重规则可配置且可保存**（默认：同一玩家只算最早一条有效留言，防刷楼）：
  - 只算最早一条（默认）
  - 全部保留（每条都参与排名/抽选）
  - 算最优一条（如点赞最高的那条）

### 步骤 3 · 拉取玩家信息
- 中台按当前配置拼 Eastblue 下载链接，后端 Playwright 自动打开 → 自动下载 xlsx → 自动读取 → 按玩家 ID 匹配。
- **配置项在界面可调且可保存**（默认值来自现有链接）：
  - `game_id`（默认 97）
  - `game_langs`（多选：en,fr,de,ja,ru,ko,es,tr,ar,it,pt,th）
  - `except_internal`（默认 1，排除内玩）
  - `search_num`（默认 1000）
  - `last_act_time`（last_one/two/three_year，默认 three）
  - `server_ids`（可选）
- 玩家信息列：玩家ID、语言、别墅等级、角色名称、角色等级、角色创建时间、服务器、历史充值总额、最后登录时间。
- 匹配不到玩家信息的 ID → 归入无效名单（标注原因「Eastblue 无记录」）。
- 首次可能需要在自动化浏览器里登一次 SSO，登录态保存后续免登。

### 步骤 4 · 语言筛选
- 多选框勾选本次活动面向的语言（默认只选 en）。**可保存**。
- 命中语言的玩家进入下一步；未命中的剔到无效名单（标注原因「语言不符」）。

### 步骤 5 · 发奖规则 + 导出
- **可叠加多个奖项**，每个奖项配置：
  - 奖项名称（如「先锋奖」「参与奖」）
  - 规则：前 N 楼留言 / 随机抽 N 名 / 点赞最高 N 名 / 回复最高 N 名
  - 人数 N
- **奖项按顺序结算，已中奖玩家不重复进入后续奖项**（先结算先锋奖前 100，剩下的才进参与奖池）。
- 随机抽选记录**随机种子**，可复现。
- 剩余的合格玩家进「参与奖 / 未中奖」sheet。

## 输出布局

一个 xlsx，多个 sheet：
- **每个特殊奖一个 sheet**（如「先锋奖」「抽选奖」），列结构与现有 xlsm 一致：留言顺序、留言时间、玩家ID、语言、别墅等级、角色名称、角色等级、角色创建时间、服务器、历史充值总额、最后登录时间。
- **参与奖 / 未中奖 sheet**：通过语言筛选但未中特殊奖的玩家。
- **无效 sheet**：无 ID / Eastblue 无记录 / 语言不符 / 重复留言，各带原因列。
- 导出前每个 sheet 可在页面预览。

## 配置持久化

除「特殊发奖规则」外，所有用户可调的配置都能保存到本地，下次启动自动带出，不用每次重填。

**可保存的配置**：
- 步骤 2 去重规则（只算最早 / 全部保留 / 算最优）
- 步骤 3 Eastblue 配置（game_id、game_langs、except_internal、search_num、last_act_time、server_ids）
- 步骤 4 语言筛选（本次面向的语言）

**不保存的**：步骤 5 的特殊发奖规则（奖项名称 / 规则类型 / 人数）——每次活动都不同，每次现配。

**机制**：
- 保存在后端本地 `config.json`（不写死代码、不进网络）。
- 支持**多套命名预设**（如「先锋活动」「复活节活动」），一键切换；其中一套可设为默认，启动即加载。
- 界面上每组配置旁有「保存 / 另存为预设 / 恢复默认」。

## 规则引擎（后端核心）

输入：筛选后的玩家名单（含留言顺序、时间、点赞数、回复数等）。
每种规则是一个纯函数，输入名单 + N，输出中奖子集：
- **前 N 楼**：按留言顺序（或留言时间）升序取前 N。
- **随机抽 N**：给定种子打乱后取前 N。
- **点赞最高 N**：按点赞数降序取前 N（并列处理策略：留言时间早者优先）。
- **回复最高 N**：按回复数降序取前 N（同上并列策略）。
奖项间用「已中奖集合」做排除，保证不重复中奖。

## 测试

- **单元测试**：
  - ID 提取：正常、多 ID、11 位数字误判、行首/冒号/空格前缀、无 ID。
  - 去重：刷楼（同 ID 多条）在三种去重策略下的结果。
  - 规则引擎：前 N / 随机（固定种子可复现）/ 点赞并列 / 回复并列 / N 超过名单长度。
  - 奖项排除：多奖项不重复中奖。
- **端到端回归**：用本次「寻找伤心松鼠」xlsm 的「原始留言」sheet 作输入，程序输出的先锋奖 / 参与奖名单应与手工结果（xlsm 里已有的两个 sheet）一致。
- **Eastblue 联调**：真实链接实测自动下载 + 读取。

## 开放实现细节（实现阶段处理，不阻塞设计）

1. Eastblue `player-management` 导出：先探测前端触发的真实导出接口/文件命名，Playwright 监听下载目录取最新 xlsx。
2. Playwright 首次 SSO 登录态持久化（user-data-dir）。
3. FB 抓取工具代码整合方式（内嵌 iframe / 复制逻辑）。

## 分期

- **Phase 1**：步骤 2 + 5 + 导出（ID 提取、规则引擎、多 sheet 导出）+ 单元测试 + xlsm 回归 —— 不依赖网络，最先落地、最易测。
- **Phase 2**：步骤 3 Eastblue Playwright 自动下载。
- **Phase 3**：步骤 1 FB 抓取整合。
- **Phase 4**：Windows 适配（视需要）。
